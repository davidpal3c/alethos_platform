# TASK-0002 — Build report + operator runbook

**Task:** Storage tier mount configuration (Phase 0B)  
**Node:** alethos-node-01  
**Scope:** `/platform`, `/data`, `/backups`, UUID `fstab`, bind mounts. **No k3s, no observability.**

---

## Goal

- Mount **platform** NVMe at `/platform`, **database** NVMe at `/data`, **backup** HDD at `/backups`, all via **UUID** in `/etc/fstab`.
- Create directory layout under `/platform` and `/data`; configure **bind mounts** from `/platform/*` to `/var/lib/*` for future k3s/observability.
- Leave boot RAID (`sda` + `sdc`, `md0`/`md1`/`md2`) **unchanged**.

---

## Live disk mapping (verify before any destructive step)

| Role | Expected on this node | **You must confirm** |
|------|------------------------|----------------------|
| Boot RAID1 | `sda` + `sdc` → `md0`/`md1`/`md2` | `lsblk -f` — do **not** format these. |
| Backup 1TB HDD | **`sdb`** (~931G) | **Not** `sdc`. |
| Platform tier | Smaller NVMe (~238–240G), often `nvme1n1` | Match size/model to WD SN530 if labeled. |
| Database tier | Larger NVMe (~477G+ if 512G class), often `nvme0n1` | If sizes differ from docs, still assign **smaller NVMe → `/platform`**, **larger → `/data`**. |

**Golden rule:** Run `lsblk -f` and `lsblk -o NAME,SIZE,MODEL` once. Only run `mkfs` / `parted` on the three devices you label **PLATFORM_NVME**, **DATA_NVME**, **BACKUP_HDD**.

# Operator runbook (execute on alethos-node-01)

### 0. Preconditions

- Logged in as sudo-capable user (e.g. `yugen0`).
- TASK-0001 verified: RAID healthy, system boots.
- **Optional:** If `/dev/sdb` has data you need, back it up; Step 3 **wipes** the backup disk partition you create.

---

### 1. Identify disks (non-destructive)

```bash
lsblk -o NAME,SIZE,MODEL,TYPE
lsblk -f
cat /proc/mdstat
```

**Record in your notes:**

- Which block device is **~238–240G NVMe** → call it **`$PLAT_DEV`** (whole disk, e.g. `/dev/nvme1n1`).
- Which is **larger NVMe** → **`$DATA_DEV`** (e.g. `/dev/nvme0n1`).
- **`$BACKUP_DEV=/dev/sdb`** — must be ~931G HDD, **not** `sda` or `sdc`.

**Stop if:** `$PLAT_DEV` or `$DATA_DEV` would be `sda`, `sdc`, or any `md*`. **Do not continue** until names match the table above.

Export for the session (replace with your actual nodes):

```bash
export PLAT_DEV=/dev/nvme1n1    # example: smaller NVMe
export DATA_DEV=/dev/nvme0n1    # example: larger NVMe
export BACKUP_DEV=/dev/sdb      # 1TB HDD
```

---

### 2. Partition each tier disk (one GPT partition per disk)

**Destructive** for empty areas on `$PLAT_DEV`, `$DATA_DEV`, `$BACKUP_DEV`. If a disk already has a single ext4 partition you want to keep, skip `mkpart` for that disk and use existing `${DEV}p1`.

For **whole-disk single partition** (typical):

```bash
# Platform NVMe
sudo wipefs -a "$PLAT_DEV" 2>/dev/null || true
sudo parted -s "$PLAT_DEV" mklabel gpt
sudo parted -s "$PLAT_DEV" mkpart platform ext4 1MiB 100%
sudo parted -s "$PLAT_DEV" set 1 name platform

# Database NVMe
sudo wipefs -a "$DATA_DEV" 2>/dev/null || true
sudo parted -s "$DATA_DEV" mklabel gpt
sudo parted -s "$DATA_DEV" mkpart data ext4 1MiB 100%
sudo parted -s "$DATA_DEV" set 1 name data

# Backup HDD
sudo wipefs -a "$BACKUP_DEV" 2>/dev/null || true
sudo parted -s "$BACKUP_DEV" mklabel gpt
sudo parted -s "$BACKUP_DEV" mkpart backups ext4 1MiB 100%
sudo parted -s "$BACKUP_DEV" set 1 name backups
```

**Partition nodes** are usually `${PLAT_DEV}p1`, `${DATA_DEV}p1`, `${BACKUP_DEV}1` → on SATA often `/dev/sdb1`. Verify:

```bash
lsblk -f "$PLAT_DEV" "$DATA_DEV" "$BACKUP_DEV"
```

---

### 3. Create ext4 and capture UUIDs

```bash
sudo mkfs.ext4 -F -L platform "${PLAT_DEV}p1"
sudo mkfs.ext4 -F -L alethos-data "${DATA_DEV}p1"
sudo mkfs.ext4 -F -L backups "${BACKUP_DEV}1"
```

```bash
sudo blkid "${PLAT_DEV}p1" "${DATA_DEV}p1" "${BACKUP_DEV}1"
```

Copy the three **UUID=** values into a scratch file. Example placeholders below:

- `UUID_PLAT=xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx`
- `UUID_DATA=...`
- `UUID_BAK=...`

---

### 4. Mount-once test (before fstab)

```bash
sudo mkdir -p /platform /data /backups
sudo mount UUID="$UUID_PLAT" /platform
sudo mount UUID="$UUID_DATA" /data
sudo mount UUID="$UUID_BAK" /backups
df -h /platform /data /backups
```

**Expected:** Three ext4 mounts with sizes matching ~240G, larger NVMe, ~931G.

```bash
sudo umount /platform /data /backups
```

---

### 5. Directory layout

```bash
sudo mkdir -p /platform/{rancher,kubelet,containerd,prometheus,loki}
sudo mkdir -p /data/{postgres,postgres_wal,redis}
sudo mkdir -p /backups
```

Bind targets must exist **before** bind mounts:

```bash
sudo mkdir -p /var/lib/rancher /var/lib/kubelet /var/lib/containerd \
  /var/lib/prometheus /var/lib/loki
```

**Ownership (baseline for later k3s):** root-owned is acceptable until k3s is installed; TASK-0003 may adjust. Optional explicit mode:

```bash
sudo chown root:root /platform /data /backups
sudo chmod 755 /platform /data /backups
```

---

### 6. `/etc/fstab` — order matters

1. Base mounts: `/platform`, `/data`, `/backups` **first**.
2. Bind mounts **after** `/platform`.

**Backup fstab:**

```bash
sudo cp -a /etc/fstab /etc/fstab.pre-task-0002
```

Append (replace UUIDs with yours):

```fstab
# TASK-0002 — storage tiers (UUID only)
UUID=<UUID_PLAT>  /platform  ext4  defaults,nofail  0  2
UUID=<UUID_DATA>  /data      ext4  defaults,nofail  0  2
UUID=<UUID_BAK>   /backups   ext4  defaults,nofail  0  2

# Bind mounts (require /platform mounted first)
/platform/rancher     /var/lib/rancher     none  bind,nofail  0  0
/platform/kubelet     /var/lib/kubelet     none  bind,nofail  0  0
/platform/containerd  /var/lib/containerd  none  bind,nofail  0  0
/platform/prometheus  /var/lib/prometheus  none  bind,nofail  0  0
/platform/loki        /var/lib/loki        none  bind,nofail  0  0
```

Validate and mount all:

```bash
sudo mount -a
sudo systemctl daemon-reload
```

---

### 7. Verification checklist

```bash
# UUIDs match fstab
lsblk -f
grep -v '^#' /etc/fstab | grep -v '^$'

# All mounts including binds
findmnt -A | grep -E 'platform|/data|backups|rancher|kubelet|containerd|prometheus|loki'

df -h /platform /data /backups /var/lib/rancher

cat /proc/mdstat
```

- [ ] `lsblk -f`: UUIDs for `nvme*p1` and `sdb1` match `/etc/fstab` lines for `/platform`, `/data`, `/backups`.
- [ ] `findmnt -A`: all five bind targets show `/platform/...` as source.
- [ ] `df -h`: capacities consistent with ~240G / larger NVMe / ~931G HDD.
- [ ] `cat /proc/mdstat`: RAID still `[UU]` on md0/md1/md2.

**Cold reboot test:**

```bash
sudo reboot
```

After login:

```bash
findmnt /platform /data /backups
findmnt /var/lib/containerd
ls -la /platform /data /backups
```

---

### 8. Rollback

If something fails **before** reboot:

```bash
sudo umount /var/lib/loki /var/lib/prometheus /var/lib/containerd \
  /var/lib/kubelet /var/lib/rancher 2>/dev/null || true
sudo umount /backups /data /platform 2>/dev/null || true
sudo cp -a /etc/fstab.pre-task-0002 /etc/fstab
sudo mount -a
```

If the system fails to boot after a bad `fstab`, boot from Ubuntu live USB, chroot or edit `/etc/fstab` on root RAID to comment TASK-0002 lines, then reboot.

To **remove** tier filesystems (nuclear — only if redoing TASK-0002): comment fstab, reboot, then `wipefs`/`parted` on the three non-boot disks only.

## Commands reference (summary)

| Step | Action |
|------|--------|
| Identify | `lsblk -f`, `lsblk -o NAME,SIZE,MODEL` |
| Partition | `parted mklabel gpt` + `mkpart ... 1MiB 100%` on `$PLAT_DEV`, `$DATA_DEV`, `$BACKUP_DEV` |
| Format | `mkfs.ext4 -L ...` on each `p1` (or `sdb1`) |
| UUID | `blkid` |
| fstab | UUID lines for `/platform`, `/data`, `/backups` then `bind` lines |
| Test | `mount -a`, `findmnt -A`, reboot |

---

## Risks

- **Wrong disk:** Formatting `sda`/`sdc` or an md member destroys the OS. Always `lsblk -f` first.
- **sdb confusion:** On this node backup is **`sdb`**, not `sdc`.
- **NVMe size vs docs:** If “database” NVMe is not 1TB in hardware, document actual size in `project-context.json`.
- **Bind mount order:** If base `/platform` fails to mount, binds must not break boot — `nofail` on ext4 helps; binds with `nofail` avoid emergency mode for missing optional paths (verify behavior on your Ubuntu version).
- **Existing data** on NVMe/HDD: full-disk partition + mkfs **erases** prior content.
