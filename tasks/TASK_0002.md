## TASK-0002 — Storage Tier Mount Configuration

Objective

Configure the storage tiers for the Alethos platform by mounting the platform,
database, and backup disks and preparing directories required by runtime
services.

This task establishes the storage layout required for platform workloads
after the base operating system and RAID1 boot tier are installed.

---

Hardware

WD PC SN530 NVMe (240GB)  
NVMe (1TB)  
SATA HDD (1TB)

Expected devices:

/dev/nvme0n1 → Platform tier  
/dev/nvme1n1 → Database tier  
/dev/sdc → Backup tier

---

Mount Architecture

Platform Tier

/dev/nvme0n1p1 → /platform

Database Tier

/dev/nvme1n1p1 → /data

Backup Tier

/dev/sdc1 → /backups

All mounts must be configured using filesystem UUIDs.

---

Platform Runtime Directories

The platform NVMe will host high-write runtime directories.

Required directories:

/platform/rancher  
/platform/kubelet  
/platform/containerd  
/platform/prometheus  
/platform/loki

Bind mounts must map these directories to the default system paths.

Bind mounts:

/platform/rancher → /var/lib/rancher  
/platform/kubelet → /var/lib/kubelet  
/platform/containerd → /var/lib/containerd  
/platform/prometheus → /var/lib/prometheus  
/platform/loki → /var/lib/loki

---

Database Directories

The database NVMe must host stateful workloads.

Required directories:

/data/postgres  
/data/postgres_wal  
/data/redis

These directories isolate database data and WAL storage from the OS disk.

---

Backup Storage

The HDD tier is reserved for backup and archive storage.

Mount point:

/backups

Intended usage:

database dumps  
snapshot exports  
log archive retention

Backup workloads must never compete with active runtime storage.

---

Constraints

Platform workloads must not run on the OS disk.

High-write runtime directories must be moved off the boot RAID.

Database storage must remain isolated from platform runtime disks.

Backup storage must exist on a separate physical device.

All persistent mounts must use UUID-based entries in `/etc/fstab`.

---

Rationale

This storage layout enforces the platform’s four failure domains:

OS  
Platform runtime  
Database  
Backups

Separating these workloads improves reliability, recoverability,
and operational safety.

The layout intentionally mirrors patterns used in production
infrastructure environments.

---

Expected Outcome

After completion of this task the system will expose the following
storage layout:

BOOT RAID

/dev/md0 → /boot  
/dev/md1 → /  
/dev/md2 → /var

PLATFORM

/dev/nvme0n1p1 → /platform

DATABASE

/dev/nvme1n1p1 → /data

BACKUPS

/dev/sdc1 → /backups

Platform runtime directories will be correctly bind-mounted
to `/var/lib` locations used by container runtimes and
observability components.

The system will now be ready for container orchestration
installation in later phases.
