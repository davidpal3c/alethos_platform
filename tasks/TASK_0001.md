# TASK-0001

Title: Ubuntu Server installation and RAID1 boot configuration

Phase: Phase 0 — Platform Foundation

Status: Ready

Priority: Critical

Dependencies: None


# Goal

Install Ubuntu Server as the base operating system for the Alethos Platform node and configure a resilient RAID1 boot configuration using the two Intel SATA SSDs.

The result must be a **headless, SSH-accessible platform node** with:

• predictable mount layout  
• RAID1 mirrored boot disks  
• stable OS environment for Kubernetes  
• disk layout aligned with platform storage design.


# Why Now

The operating system and disk layout define the **foundation for all subsequent infrastructure work**.

If this layer is incorrect, it becomes difficult to:

• isolate platform workloads  
• maintain disk reliability  
• recover from failures  
• preserve storage tier boundaries.

## Context Summary for Builder (TASK-0001)

Hardware:

- Lenovo ThinkStation P510  
- Intel SSD Pro 1500 (180GB)  
- Intel SSD 520 (180GB)  

Boot architecture:

- UEFI boot  
- mdadm RAID1  
- Dual EFI partitions (one on each SSD)  

RAID layout:

- /dev/md0 → /boot  
- /dev/md1 → /  
- /dev/md2 → /var  

Partition scheme (per SSD):

- 512MB EFI System Partition  
- 2GB RAID member for /boot  
- 50GB RAID member for /  
- Remaining space RAID member for /var  

Rationale:

- Boot resilience  
- OS stability  
- Log isolation from root filesystem  
- Preparation for future platform workloads on separate tiers  

Constraints:

- No platform workloads on the OS RAID tier  
- k3s runtime and high-churn data must be moved to the platform NVMe (/platform) in later tasks

# Why /var is Separate

/var is isolated so that:

• logs  
• container metadata  
• system services  

cannot fill the root filesystem and brick the OS.


# OS Installation Requirements

Distribution:

Ubuntu Server LTS


Installation mode:

Minimal installation.


During installation:

• create RAID arrays using mdadm  
• assign mountpoints as defined above  
• enable OpenSSH server


User requirements:

Create admin user with sudo access.


# Boot Resilience Requirements

After installation:

GRUB must be installed on both disks.

Example:

grub-install /dev/sda  
grub-install /dev/sdb

EFI partition must be duplicated on both disks.

Future sync strategy:

rsync EFI partition contents periodically.

# Acceptance Criteria

The system is considered correctly installed when:

• server boots successfully  
• RAID arrays show healthy state  
• mount layout matches design  
• SSH login works  
• disks are monitored.


# Verification Steps

Run:

lsblk

Confirm RAID devices exist.


Check RAID status:

cat /proc/mdstat


Confirm mounts:

df -h


Check RAID metadata:

mdadm --detail /dev/md0
mdadm --detail /dev/md1
mdadm --detail /dev/md2


Reboot test:

System must reboot cleanly and remain accessible via SSH.


# Failure Conditions

Task fails if:

• RAID arrays degraded after install  
• mountpoints incorrect  
• boot works only from one disk  
• OS disk usage already near capacity.


# Risks

Mixed-model SSD RAID.

Risk mitigation:

Mirror size determined by smallest drive.


Potential BIOS boot order issue.

Mitigation:

Ensure both disks are bootable.
